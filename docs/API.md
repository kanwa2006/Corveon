# Corveon — API Contract (REST + SSE)

The canonical machine-readable contract is the OpenAPI schema generated by FastAPI at
`/openapi.json` (interactive docs at `/docs`). This file is the narrative companion and the
source of truth **until** the app is generated; frontend types are generated from the OpenAPI
schema and kept honest by contract tests (schemathesis).

## Conventions
- JSON everywhere. Base path: `/api/v1`.
- Auth: `Authorization: Bearer <access_token>`.
- Uniform error model: `{ "error_code", "message", "details", "trace_id" }`.
- Status codes: `200 201 202 204 400 401 403 404 409 422 429 500`.
- Long/streaming work uses **SSE served by the FastAPI backend** (never Vercel serverless, §23.3).
- Every response that runs the orchestrator carries a `routing_trace` object.

## Auth
| Method | Path | Result |
|---|---|---|
| POST | `/auth/register` | `201 {user}` |
| POST | `/auth/login` | `200 {access, refresh}` |
| POST | `/auth/refresh` | `200 {access}` |
| POST | `/auth/logout` | `204` |
| GET | `/auth/me` | `200 {user}` — added during Week 1 implementation: the frontend needs a way to fetch the authenticated user's profile, which `/login` (tokens only) doesn't provide. Requires a valid bearer access token. |
| POST | `/auth/stream-ticket` | `200 {ticket}` — added during Week 1 (ADR-0016): a 60-second single-purpose credential the browser passes as `?ticket=` to open the two SSE endpoints below directly against the backend (ADR-0007), since the httpOnly session cookie can't cross origins and `EventSource` can't attach a custom header. Requires a valid bearer access token to mint. |
| POST | `/auth/sso/start` `{email}` | `200 {redirect_url}` — added by ADR-0025 (Enterprise SSO, OIDC only). Looks up the org's SSO config by the email's domain; `404` if none configured or inactive. Unauthenticated (the user isn't logged in yet). |
| GET | `/auth/sso/callback?code=&state=` | `200 {access, refresh}` — the IdP redirects the browser here after authentication. Verifies `state` (single-use, Redis, 300s TTL), exchanges `code` for tokens (PKCE), verifies the `id_token` via the IdP's JWKS, then JIT-provisions or looks up the user by email and mints a session identical in shape to `/auth/login`. `403` if an existing user with that email belongs to a different org (never silently reassigns `org_id`); `401` on an invalid/expired/replayed `state` or an inactive user. |

## Messages / AI (SSE streaming)
| Method | Path | Result |
|---|---|---|
| POST | `/chats/{id}/messages` | `202` + SSE — `token` events (text deltas), a final `done` event (`{message_id, routing_trace}`), or an `error` event (`provider_unavailable`, degraded mode, ADR-0006). Implemented Week 1: the browser connects directly with a stream ticket (`?ticket=`, ADR-0016), not the session cookie. `routing_trace.path` ∈ `{fast_path, pure_llm, rag_grounded, rag_no_match, rag_public_evidence}`; `retrieved_chunks[]` (uploaded-document citations) and `public_evidence[]` (`{source, title, url, identifier, snippet, published_date}`, public-source citations — populated only on `rag_public_evidence`, ADR-0021) are always both present, kept as separate arrays since they're different trust levels. In `DEPLOYMENT_MODE=ollama_only` (ADR-0024), `routing_trace.provider` is always `ollama` and `routing_trace.path` never resolves to `rag_public_evidence` (no public connectors are registered). |
| POST | `/chats/{id}/messages/{mid}/regenerate` | `202` + SSE — planned (Month 1+), not yet implemented |
| POST | `/chats/{id}/messages/{mid}/continue` | `202` + SSE — planned (Month 1+), not yet implemented |
| GET | `/chats/{id}/messages` | `200 [message]` |

## Documents / uploads
Implemented Week 1. PDF only (extension + MIME + magic-byte checked); parsing/chunking/embedding
runs in the ARQ ingestion worker (`app/workers/tasks.py`), never inline in the request.
| Method | Path | Result |
|---|---|---|
| POST | `/chats/{id}/documents` (multipart) | `202 {job_id}` |
| GET | `/jobs/{job_id}` | `200 {status, progress_stage, error}` |
| GET | `/jobs/{job_id}/events` | SSE `stage` events (each carrying the current `JobPublic`); browser connects directly with a stream ticket (ADR-0016), same as messages |
| GET | `/chats/{id}/documents` | `200 [document]` |
| DELETE | `/documents/{id}` | `204` — rejects with `409` while the document is mid-ingestion (`status: processing`) to avoid racing the worker's own writes |

## Search
Implemented Week 1. Backed by a pluggable vector store — pgvector by default, or Qdrant when
`VECTOR_STORE=qdrant` (ADR-0022) — transparent to this contract either way. Reads from the Postgres
read replica when `DATABASE_READ_REPLICA_URL` is configured, the primary otherwise (ADR-0023) —
also transparent to this contract.
| Method | Path | Result |
|---|---|---|
| POST | `/chats/{id}/search` | `200 [hit]` — semantic, **in-chat only**; filters by both `chat_id` and embedding `model_id` (ADR-0008) |

## Evidence & medication
Evidence Verification implemented Month 3 (`app/evidence/`). Extracts the independently-verifiable
claims from one existing message, retrieves evidence per claim from this chat's own uploaded
documents plus six public connectors (PubMed, DailyMed, openFDA, ClinicalTrials.gov, MeSH,
RxNorm), then classifies source, scores confidence, and verifies each citation — see
`docs/ARCHITECTURE.md` §Evidence Verification Engine.

Medication-Safety Engine implemented Month 6-12 (`app/medication/`): free-text medication parsing,
RxNorm normalization, and deterministic drug-drug interaction detection (Phase 1 — DDInter 2.0 +
openFDA label fallback, ADR-0004); deterministic renal-dosing threshold checks (Phase 2 —
Cockcroft-Gault CrCl + 2021 race-free CKD-EPI eGFR, ADR-0005); deterministic potentially-
inappropriate-prescribing screening (AGS Beers Criteria 2023 + STOPP/START v3) and
medication-discrepancy classification, plus an optional guardrail-checked LLM narrative on those two
finding types (Phase 3, ADR-0019/ADR-0020) — see `docs/ARCHITECTURE.md` §Medication-Safety Engine.

| Method | Path | Result |
|---|---|---|
| POST | `/chats/{id}/verify` `{message_id}` | `202` + SSE — a `claim` event per completed, already-scored claim (`{id, ordinal, text, source_class, confidence_score, confidence_rationale, flags[], citations[]}`), a final `done` event (`{verification_id, status}`), or an `error` event (`provider_unavailable` \| `budget_exceeded`, degraded mode, ADR-0006). Same shape as `POST /chats/{id}/messages`; the browser connects directly with a stream ticket (`?ticket=`, ADR-0016). |
| POST | `/chats/{id}/medications/analyze` `{raw_text, age_years?, weight_kg?, sex?, serum_creatinine_mg_dl?, height_cm?, conditions?, previous_raw_text?}` | `202` + SSE — a `medication` event per parsed, RxNorm-normalized, already-persisted current-list medication (`{id, raw_text, name, rxcui, dose, route, frequency}`); a `previous_medication` event of the same shape per previous-list medication (only when `previous_raw_text` is supplied); an `interaction` event per DDI finding (`{id, medication_a_id, medication_b_id, severity, source, rule_id, explanation, provenance}`); a `renal` event per renal-dosing threshold finding (`{id, medication_id, crcl_ml_min, egfr_ml_min, threshold_ml_min, severity, rule_id, explanation}` — only when `weight_kg`/`sex`/`serum_creatinine_mg_dl`/`height_cm` are all supplied along with `age_years`; the four renal-only fields are all-or-nothing, a partial set is a `422`); a `pip` event per PIP finding (`{id, medication_id, source, direction, severity, rule_id, drug_names[], matched_condition, explanation, narrative}` — only when `age_years` is supplied and the patient is ≥65; `medication_id` is null for a `start_consider` finding, which flags a medication *absent* from the current list); a `discrepancy` event per list-diff finding (`{id, kind, current_medication_id, previous_medication_id, rule_id, explanation, narrative, provenance}` — only when `previous_raw_text` is supplied); a final `done` event (`{status}`); or an `error` event (`provider_unavailable` \| `budget_exceeded`, degraded mode). Same SSE shape and stream-ticket bridge as `/verify`. `severity` ∈ `{major, moderate, minor, unclassified}`; `source` (interaction) ∈ `{ddinter, openfda_label, calculated}`; `source` (pip) ∈ `{beers_2023, stopp_v3, start_v3}`; `direction` ∈ `{avoid, start_consider}`; `kind` ∈ `{added, omitted, dose_changed, frequency_changed}`. `narrative` (`pip`/`discrepancy` only) is a guardrail-checked plain-language rendering of `explanation` (ADR-0020) — null when unavailable or it failed the grounding check; `explanation` is always present regardless. |

## Org SSO (ADR-0025)
`org-admin`/`superadmin` only, always scoped to the caller's own `org_id` (never accepted from the
request body). Backed by the `/settings/sso` admin page as well as direct API use.
| Method | Path | Result |
|---|---|---|
| POST | `/org/sso-config` `{issuer, client_id, client_secret, email_domain}` | `201 {config}` — upserts the org's SSO config; `client_secret` is encrypted at rest (Fernet, `SSO_CONFIG_ENCRYPTION_KEY`) and never returned by any endpoint, same posture as a password hash |
| GET | `/org/sso-config` | `200 {config}` \| `404` if none configured |
| DELETE | `/org/sso-config` | `204` |

## Notes
- `source_class` ∈ `{uploaded_document, verified_public, org_trusted, ai_reasoning,
  conflicting_insufficient}`. `org_trusted` is a real, reserved classification — no claim is
  tagged with it until the org-trusted-sources subsystem (still planned) exists to produce it.
- Every content endpoint authorizes on **both** user identity and resource ownership; the active
  `chat_id` scopes all reads (per-chat isolation, ADR-adjacent §10.2).
- Rate limiting returns `429` with `Retry-After`; providers are load-balanced/failed-over behind
  the scenes (§5) — clients never select a provider.
