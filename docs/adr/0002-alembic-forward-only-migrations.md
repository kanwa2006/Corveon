- **Status:** Accepted

## Context
"Migration conflicts" is a named past failure category. We need deterministic, reviewable schema
evolution with SQLAlchemy 2.0.

## Decision
Use **Alembic**: one migration per schema change; **forward-only in production**; no auto-generated
migration merged without human review. CI runs a check asserting that a fresh `--autogenerate`
against `head` produces an **empty diff** (models and migrations are in sync). Migrations that add
content-bearing tables must also add the corresponding Row-Level Security policy.

## Consequences
- Schema drift between ORM models and the database is caught in CI, not production.
- Rollbacks are done by rolling **forward** with a new migration; downgrades are written and tested
  but never run in production.
- Slight discipline cost: every model change needs an accompanying, reviewed migration.

## Alternatives considered
- **Auto-apply autogenerated migrations:** fast but exactly how "migration conflicts" happen.
- **No migration tool / manual SQL:** not reproducible or reviewable.
